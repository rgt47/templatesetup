\section{Introduction}\label{introduction}

I did not really appreciate how fragile my shell configuration was until
I tried to replicate my macOS development environment on a second
laptop. Every alias, every shell function, every launchd scheduled job,
and every editor setting lived inside
\texttt{\textasciitilde{}/Library/CloudStorage/Dropbox/dotfiles/}, a
directory with no \texttt{.git} folder, no version history, no diff
capability, and no rollback mechanism. The only `backup' was Dropbox's
version history, which I discovered was inadequate when a
\texttt{sed\ -i} command on a cloud-mounted path truncated a 546-line
script to zero bytes.

The design for a better arrangement already existed. Post 24,
`\href{https://focusonr.org/posts/24-setupdotfilesongithub/}{Creating a
GitHub Dotfiles Repository}', laid out the correct architecture: a
standalone \texttt{\textasciitilde{}/dotfiles} git repository outside
any cloud-sync path, an \texttt{install.sh} that creates symlinks with
timestamped backups, platform detection via \texttt{\$OSTYPE}, a
\texttt{Brewfile}, and an \texttt{.env.local} pattern for secrets. That
post was written and published in February 2026 but never executed
against the actual configuration.

This post documents the audit that identified three concrete blockers
preventing multi-laptop deployment, and the migration procedure that
resolves them. It extends post 24's design with five additions: launchd
plist parameterisation, a pipx tool manifest, a vim plugin-manager
strategy, an explicit sensitive-files inventory, and migration steps
from the current half-installed state.

\subsection{Motivations}\label{motivations}

\begin{itemize}
\tightlist
\item
  The \texttt{tn} incident (a \texttt{sed\ -i.bak} against a Dropbox
  path truncated a production script to zero bytes) demonstrated that
  cloud-mounted paths are not safe working trees; git is the correct
  synchroniser.
\item
  Setting up a second MacBook without a deployment mechanism means
  manually recreating months of accumulated configuration from memory,
  which is both error-prone and time-consuming.
\item
  Hardcoded \texttt{\textasciitilde{}/Library/CloudStorage/Dropbox/...}
  paths in scripts and launchd plists break silently on any machine
  where Dropbox is absent, uses a different account, or is installed at
  a legacy path.
\item
  Sensitive files (\texttt{\textasciitilde{}/.aws},
  \texttt{\textasciitilde{}/.ssh}, \texttt{\textasciitilde{}/.gnupg})
  live adjacent to tracked dotfiles with no formal inventory and no
  explicit decision about what belongs in git versus what must remain
  local.
\item
  Understanding the gap between a written architecture and an executed
  one is itself useful: post 24's design was sound but several category
  errors (vim plugin bundles treated as config, runtime caches mixed
  with user config) were only visible during an actual audit.
\end{itemize}

\subsection{Objectives}\label{objectives}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Identify every file that must be tracked, excluded, or handled via a
  separate secrets mechanism, producing an explicit inventory.
\item
  Create \texttt{\textasciitilde{}/dotfiles} as a git repository outside
  any cloud-sync path, structured per post 24's layout, and push it to a
  private GitHub repository.
\item
  Write and test \texttt{install.sh} with \texttt{-\/-dry-run} support,
  launchd \texttt{\_\_USER\_\_} substitution, and per-file symlinks for
  \texttt{\textasciitilde{}/bin}.
\item
  Verify that a clean bootstrap (on a fresh user account or VM) runs
  without hardcoded-path failures, undefined variable errors, or leaked
  secrets.
\end{enumerate}

\section{What is a Dotfiles
Repository?}\label{what-is-a-dotfiles-repository}

A dotfiles repository is a version-controlled directory that stores
shell configuration files, editor settings, and system scripts in a form
that can be deployed to any new machine with a single command. Think of
it as the source code for your development environment: just as
application code can be cloned and built on a fresh server, a dotfiles
repository can be cloned and installed on a fresh laptop.

The concrete benefit is reproducibility. Without a dotfiles repository,
two machines diverge the moment you customise one of them. With one, the
command
\texttt{git\ clone\ git@github.com:rgt47/dotfiles\ \textasciitilde{}/dotfiles\ \&\&\ cd\ \textasciitilde{}/dotfiles\ \&\&\ ./install.sh}
brings a new laptop to the same state as the primary machine, minus
secrets that must be installed by a separate mechanism.

\section{Prerequisites}\label{prerequisites}

This post assumes:

\begin{itemize}
\tightlist
\item
  \textbf{Operating system:} macOS 13+ (Ventura or later); the
  architecture extends to Ubuntu 22.04+ with minor adjustments
\item
  \textbf{Hardware:} Apple Silicon or Intel Mac (the scripts are
  architecture-agnostic)
\item
  \textbf{Already installed:} git, Homebrew, zsh 5.9+, pipx (for Python
  CLI tools)
\item
  \textbf{Background knowledge:} comfort editing dotfiles, writing bash
  scripts, and running shell commands; familiarity with post 24's design
\item
  \textbf{GitHub account:} with SSH access configured (the repository
  must be private; never use a public repository for dotfiles)
\item
  \textbf{Time required:} 3-4 hours for the initial migration; 30
  minutes to bootstrap subsequent laptops
\end{itemize}

If you are starting from scratch rather than migrating, post 24's
step-by-step walkthrough is the right entry point. This post assumes the
Dropbox-based status quo described above.

\section{Audit Findings: Three
Blockers}\label{audit-findings-three-blockers}

Before writing any code, an audit of \texttt{\textasciitilde{}/bin},
\texttt{\textasciitilde{}}, \texttt{\textasciitilde{}/.config}, and
\texttt{\textasciitilde{}/.local} identified three issues that must be
resolved before dotfiles can safely be exported to a second machine.

\subsection{Blocker 1: No Version
Control}\label{blocker-1-no-version-control}

\texttt{\textasciitilde{}/Library/CloudStorage/Dropbox/dotfiles/} has no
\texttt{.git} directory. The entire shell configuration is a
Dropbox-synced tree with no history, no diff capability, and no
rollback. Dropbox version history is the only recovery path, and as the
\texttt{tn} incident showed, that recovery is not guaranteed for
in-place edits.

The repository must live at \texttt{\textasciitilde{}/dotfiles}, outside
any cloud-sync path. Git writes constantly to \texttt{.git/index},
\texttt{.git/HEAD}, and \texttt{.git/refs/...}; a cloud provider syncing
those writes simultaneously causes race conditions that corrupt the
index and can truncate files to zero bytes. The
\texttt{\textasciitilde{}/dotfiles} directory is local to each machine;
git itself (with GitHub as the remote) is the synchroniser.

\subsection{Blocker 2: Hardcoded Dropbox
Paths}\label{blocker-2-hardcoded-dropbox-paths}

Every script in \texttt{\textasciitilde{}/bin/launchd/}, the launchd
plists, the \texttt{cdpath} in \texttt{.zshrc}, and the \texttt{vz}
alias all hardcode
\texttt{\textasciitilde{}/Library/CloudStorage/Dropbox/...}. On a second
laptop where Dropbox is absent, uses a different account, or is
installed at the legacy path (\texttt{\textasciitilde{}/Dropbox}), every
reference fails silently.

The fix is a single environment variable sourced from
\texttt{\textasciitilde{}/.zshenv}:

\begin{Shaded}
\begin{Highlighting}[]
\CommentTok{\# \textasciitilde{}/.zshenv}
\BuiltInTok{export} \VariableTok{DOTFILES}\OperatorTok{=}\StringTok{"}\VariableTok{$\{DOTFILES}\OperatorTok{:{-}}\VariableTok{$HOME}\NormalTok{/dotfiles}\VariableTok{\}}\StringTok{"}
\BuiltInTok{export} \VariableTok{PRJ\_ROOT}\OperatorTok{=}\StringTok{"}\VariableTok{$\{PRJ\_ROOT}\OperatorTok{:{-}}\VariableTok{$HOME}\NormalTok{/Dropbox/prj}\VariableTok{\}}\StringTok{"}
\KeywordTok{[[} \OtherTok{{-}d} \StringTok{"}\VariableTok{$PRJ\_ROOT}\StringTok{"} \KeywordTok{]]} \KeywordTok{||} \DataTypeTok{\textbackslash{}}
    \BuiltInTok{export} \VariableTok{PRJ\_ROOT}\OperatorTok{=}\StringTok{"}\VariableTok{$HOME}\StringTok{/Library/CloudStorage/Dropbox/prj"}
\end{Highlighting}
\end{Shaded}

Every script and config then references \texttt{\$DOTFILES} and
\texttt{\$PRJ\_ROOT} rather than literal paths.

\subsection{Blocker 3: Sensitive Files Without an
Inventory}\label{blocker-3-sensitive-files-without-an-inventory}

Confirmed sensitive items in \texttt{\textasciitilde{}}:

\begin{verbatim}
~/.aws      AWS credentials (plaintext)
~/.docker   registry tokens
~/.env      project secrets
~/.gnupg    private keys
~/.npmrc    npm registry token
~/.password-store  pass database (encrypted)
~/.ssh      SSH private keys
~/.vpn      VPN configuration
\end{verbatim}

These must never be committed to git, even in a private repository,
because they appear in every cloned working tree and persist in history
forever. A \texttt{.gitignore} written before the first
\texttt{git\ add} is the only reliable control; after the first commit,
sensitive data requires a full \texttt{git\ filter-repo} rewrite and
credential rotation.

\section{Things to Watch Out For}\label{things-to-watch-out-for}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\item
  \textbf{Write \texttt{.gitignore} before \texttt{git\ init}, not
  after.} If any sensitive file is committed even once, it must be
  removed with \texttt{git\ filter-repo} and every credential it
  contained must be rotated. There is no lighter-weight remediation.
\item
  \textbf{Do not place the dotfiles repo inside Dropbox, iCloud, or
  Google Drive.} Git writes to \texttt{.git/index}, \texttt{.git/HEAD},
  and ref files continuously; a cloud provider racing to sync those
  writes can corrupt the index or truncate files to zero bytes. This is
  the same failure mode as the \texttt{tn} incident (2026-05-03).
\item
  \textbf{The launchd \texttt{\_\_USER\_\_} substitution writes to
  \texttt{\textasciitilde{}/Library/LaunchAgents/}, not back to the
  repo.} If you edit a plist in
  \texttt{\textasciitilde{}/dotfiles/launchd/} and re-run
  \texttt{install.sh}, the substituted file is overwritten in place.
  Never \texttt{git\ add} the file from
  \texttt{\textasciitilde{}/Library/LaunchAgents/}; always edit the
  template in \texttt{\textasciitilde{}/dotfiles/launchd/}.
\item
  \textbf{vim plugin bundle directories must not be tracked.} The 22M
  \texttt{\textasciitilde{}/.config/vim/} directory is almost entirely
  downloaded plugin code managed by vim-plug or Vundle. Track only the
  \texttt{vimrc} and the plugin manifest; the installer should run the
  plugin manager on first boot. The same applies to
  \texttt{\textasciitilde{}/.config/coc/} (1.6M of language-server
  cache). Add both to \texttt{.gitignore} before the first commit.
\item
  \textbf{\texttt{\textasciitilde{}/bin} must be a real directory
  populated by per-file symlinks, not a symlink to a Dropbox directory.}
  A top-level
  \texttt{\textasciitilde{}/bin\ -\textgreater{}\ \textasciitilde{}/Library/CloudStorage/Dropbox/bin}
  symlink dangles on any machine without Dropbox. The installer creates
  individual symlinks from
  \texttt{\textasciitilde{}/dotfiles/bin/\textless{}script\textgreater{}}
  to \texttt{\textasciitilde{}/bin/\textless{}script\textgreater{}}.
\item
  \textbf{Test \texttt{install.sh\ -\/-dry-run} on a non-primary user
  account before running it on a second laptop.} The first dry-run on a
  fresh environment will surface hardcoded paths and assumptions
  invisible on the primary machine. Identify and fix them before the
  real bootstrap.
\end{enumerate}

\section{What Did We Learn?}\label{what-did-we-learn}

\subsection{Lessons Learnt}\label{lessons-learnt}

\textbf{Conceptual Understanding:}

\begin{itemize}
\tightlist
\item
  A dotfiles repository is the source code for a development
  environment. Like application code, it should live in git, be tested
  before deployment, and have a migration path for breaking changes.
\item
  Cloud-sync and git are fundamentally different synchronisation
  mechanisms with conflicting write patterns. Using cloud storage as a
  substitute for git version control is not a safe design; it provides
  neither the branching, diffing, nor rollback capabilities of git,
  while adding race-condition risk.
\item
  The distance between a written architecture and an executed one
  accumulates debt. Post 24's design was correct, but the gap between
  the written plan and the actual filesystem meant three months of
  configuration changes had no version history.
\item
  An explicit inventory of sensitive files, written before the first
  \texttt{git\ add}, is a one-time cost that prevents a very expensive
  recovery. Credential rotation after an accidental commit is not
  hypothetical; it has happened to experienced engineers.
\end{itemize}

\textbf{Technical Skills:}

\begin{itemize}
\tightlist
\item
  The \texttt{link\_file} pattern (back up existing files with a
  timestamp, then create a symlink) makes \texttt{install.sh} idempotent
  and re-runnable without destroying local changes.
\item
  The \texttt{\_\_USER\_\_} substitution pattern for launchd plists is a
  clean solution to the constraint that macOS \texttt{launchd} does not
  expand \texttt{\textasciitilde{}} in plist files.
\item
  A \texttt{-\/-dry-run} flag in shell scripts (print every action
  without executing) is low-cost to implement and essential for
  validating an installer on a production machine before committing.
\item
  Separating package manifests (\texttt{Brewfile},
  \texttt{pipx-tools.txt}, \texttt{nvim-plugins.lua}) from the install
  script means adding a new tool requires one line in a manifest file,
  not an edit to the installer logic.
\end{itemize}

\textbf{Gotchas and Pitfalls:}

\begin{itemize}
\tightlist
\item
  The order of operations matters: \texttt{.gitignore} before
  \texttt{git\ init}, \texttt{git\ status} review before
  \texttt{git\ add}, \texttt{-\/-dry-run} before the real run. Any of
  these steps skipped creates a problem that is harder to fix than to
  prevent.
\item
  Runtime caches and downloaded plugin code are frequently mixed with
  user configuration in \texttt{\textasciitilde{}/.config}. The rule is:
  if a directory would be regenerated by running a tool, it is not user
  configuration and must not be tracked in git.
\item
  \texttt{\textasciitilde{}/bin} as a Dropbox symlink works on the
  primary machine but produces a dangling symlink on any other machine.
  The per-file symlink pattern is more verbose but correct across
  environments.
\item
  The \texttt{rclone} configuration file
  (\texttt{\textasciitilde{}/.config/rclone/rclone.conf}) contains OAuth
  tokens and must be scrubbed of credentials before it is committed,
  even to a private repository.
\end{itemize}

\subsection{Limitations}\label{limitations}

\begin{itemize}
\tightlist
\item
  \textbf{Single-user scope:} \texttt{install.sh} deploys to
  \texttt{\$HOME} and makes no attempt to set up system-level
  configuration (\texttt{/etc/}, \texttt{/usr/local/}, or system-wide
  launchd daemons).
\item
  \textbf{macOS-centric launchd:} the launchd plist handling is macOS
  only. A Linux equivalent using systemd user units would require a
  parallel code path that this version does not provide.
\item
  \textbf{No conflict resolution:} if \texttt{install.sh} is run on a
  machine that already has dotfiles managed by a different tool (e.g.,
  chezmoi or GNU Stow), the backup-and-replace strategy may interfere
  with that tool's state.
\item
  \textbf{Secrets workflow is manual:} the installer prints a reminder
  to restore \texttt{\textasciitilde{}/.aws},
  \texttt{\textasciitilde{}/.ssh}, and
  \texttt{\textasciitilde{}/.gnupg}, but provides no automation. On a
  new laptop, secret restoration remains a manual step that must be
  performed correctly.
\item
  \textbf{Plugin manifest not yet complete:} the vim plugin manifest
  (\texttt{editors/vimrc} + a \texttt{packages/nvim-plugins.lua}
  manifest) is planned but not yet written. The 22M plugin bundle is
  excluded from git, which means the first \texttt{vim} launch on a new
  machine will require a manual \texttt{:PlugInstall}.
\end{itemize}

\subsection{Opportunities for
Improvement}\label{opportunities-for-improvement}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Add a \texttt{bootstrap.sh} that installs Homebrew and git before
  cloning the dotfiles repo, making the first command on a truly fresh
  laptop a single \texttt{curl\ \textbar{}\ bash} invocation (with
  appropriate scrutiny of what that script does).
\item
  Write a systemd user-unit equivalent of the launchd plist section,
  conditioned on \texttt{{[}{[}\ "\$OSTYPE"\ ==\ "linux-gnu"*\ {]}{]}},
  so the same dotfiles repository works on Ubuntu development VMs.
\item
  Add GitHub Actions CI that runs \texttt{install.sh\ -\/-dry-run} and
  \texttt{make\ lint} on macOS and Ubuntu runners after every push,
  catching cross-platform breakage before a laptop bootstrap.
\item
  Complete the vim plugin manifest and add a post-install hook to
  \texttt{install.sh} that runs \texttt{:PlugInstall} in headless vim,
  making the editor ready without manual intervention.
\item
  Evaluate \texttt{git-crypt} for encrypting a small set of
  near-sensitive configs (rclone config, mbsyncrc) so they can live in
  the repository without requiring manual scrubbing of credentials.
\end{enumerate}

\section{Wrapping Up}\label{wrapping-up}

The gap between a correct architecture and an executed one is easy to
accumulate and expensive to close. Post 24 described the right design
for a portable dotfiles repository in February 2026; this post documents
the audit that revealed why it had not been executed, identifies the
three blockers (no version control, hardcoded Dropbox paths, and
sensitive files without an inventory), and provides the concrete scripts
and migration procedure.

The highest-leverage change is also the simplest: \texttt{git\ init} in
a directory outside cloud sync, with a \texttt{.gitignore} written
before the first commit. Everything else, the \texttt{install.sh}
parameterisation, the launchd \texttt{\_\_USER\_\_} substitution, the
plugin-manifest strategy, builds on that foundation.

\textbf{Main takeaways:}

\begin{itemize}
\tightlist
\item
  Cloud sync is not version control. A dotfiles repository outside any
  cloud-mounted path, with GitHub as the remote, is the correct design.
\item
  Write \texttt{.gitignore} before \texttt{git\ init}. This is not a
  best practice; it is a prerequisite. One sensitive commit requires a
  full history rewrite and credential rotation.
\item
  \texttt{install.sh\ -\/-dry-run} before every real install. The cost
  of reviewing the plan is ten seconds; the cost of an unexpected
  overwrite can be an afternoon.
\item
  Per-file symlinks in \texttt{\textasciitilde{}/bin} are more verbose
  than a top-level symlink but are the only pattern that works across
  machines with different Dropbox configurations.
\end{itemize}
